# Parallel Review Pipeline

You are an orchestrator. Your job is to launch review agents, collect their reports, and synthesise the results. You do NOT review code yourself — you delegate and summarise.

## Step 0: Determine Topology

Check `$ARGUMENTS` for the keyword `ring` or `star`.

- If `ring` → go to **Ring Topology**
- If `star` or no topology specified → go to **Star Topology**

Any remaining arguments (after removing `ring`/`star`) are the **scope** passed to each agent.

---

## Step 0.5: Pre-flight Context Brief

Before launching ANY review agents, generate a context brief at `/tmp/review-context.md`. This eliminates redundant file reads across all reviewers.

### Option A: Script (preferred — zero LLM cost)

Run the extraction script via Bash:

```bash
./scripts/review-brief.sh
```

This reads ~25 key files and writes a structured brief with:
- Full docs (ARCHITECTURE.md, SECURITY.md — already compressed by humans)
- Full declarations (schema.yaml, navigation.yaml, prisma schema)
- Exports + function signatures for all source files (not full contents)
- Config files (Dockerfile, docker-compose, CI)
- Test file inventory

**No LLM tokens spent.** The script runs in <1 second.

### Option B: LLM-compressed brief (when you need a tighter summary)

If the script output is too large for the agents' context, spawn ONE agent with `model: "haiku"` to compress it:

```
First, read ~/.claude/AGENT.md for instructions.

Read /tmp/review-context.md (generated by scripts/review-brief.sh).
Compress it to under 400 lines: summarise the docs sections, keep source
exports and declarations verbatim, drop boilerplate from config files.
Write the compressed version back to /tmp/review-context.md.
```

This costs ~5K tokens on haiku (~$0.005) vs ~40K tokens on sonnet (~$0.24) for a from-scratch read.

### When to use which

- **Option A** for routine reviews — the script output (~2300 lines) is fine for sonnet-class reviewers
- **Option A + B** when reviewer agents are haiku — compress to stay within their effective attention span

**CRITICAL:** The brief must exist at `/tmp/review-context.md` before launching any review agents.

## Star Topology (default — fast, independent)

All five agents run in parallel. No agent sees another's findings. Each agent reads the pre-flight brief instead of exploring the codebase independently.

### Launch all five in a single message

Each agent's prompt:

```
First, read ~/.claude/AGENT.md for instructions.

```

## Ring Topology (deep — sequential with context passing)

Agents run one at a time. Each receives the accumulated findings from all previous agents. The architect runs twice: once at the start (fresh structural analysis) and once at the end (root-cause analysis of all findings).

```
Arch₁ → Style → Security → Compliance → Production → Arch₂ → Report
```

### Progressive Disclosure

The pre-flight brief gives every agent a compressed summary of the whole codebase. But as agents run, they discover which files are actually problematic. The orchestrator's job between agents is to **extract focus areas** — telling the next agent which files to read directly rather than trusting the brief.

**Between each agent run, the orchestrator MUST:**

1. Read the agent's report for `file:line` references and flagged concerns
2. Build a `{focus_areas}` block listing:
   - File path + line range that was flagged
   - One-line reason (e.g., "architect: leaky abstraction in role check")
   - Instruction: "Read this file directly — the brief summary is insufficient"
3. Inject this block into the next agent's prompt (see template below)

The focus areas accumulate: agent_3 gets focus areas from agents 1+2, agent_4 gets 1+2+3, etc. This means later agents read MORE source files — but only the files that earlier agents identified as suspicious. This is the opposite of the star topology where every agent reads everything.

**Example focus_areas block:**
```
Files flagged by prior reviewers — read these directly, do not rely on the brief:
- src/lib/auth.ts:45-80 — architect: role check doesn't cover nurse sub-routes
- src/app/api/ai/route.ts:12-35 — architect: schema description leaks field names to Gemini
- src/lib/repository.ts:120-145 — style: repeated try/catch pattern across CRUD methods
```

## Synthesise

Once all agents have returned, produce a single report:

### Dashboard

| Reviewer | Top Finding |
|----------|-------------|
| Architect | ... |
| Style | ... |
| Security | ... |
| Compliance | ... |
| Production Ready | ... |
| Architect (root causes) | ... |  ← ring only

### Cross-Cutting Themes

Findings flagged by multiple reviewers. In ring mode, the architect's second pass identifies these explicitly — highlight architectural root causes that would fix multiple downstream findings.

### Prioritised Action Items

Merge and deduplicate all findings into one list:

1. **Critical** — blocks deployment
2. **High** — fix before shipping
3. **Medium** — fix soon
4. **Low** — defence-in-depth

For each item include:
- Which reviewer(s) flagged it
- `file:line` reference
- Recommended fix
- Effort (S/M/L)
- **Files to read:** exact list of files an implementation agent needs as context
- **Files to edit:** exact list of files that need changes

In ring mode, mark items that the architect's second pass identified as symptoms of a deeper architectural issue — fixing the root cause may resolve several items at once.

### What's Working Well

Collect passed checks and praise from all reports.

## Briefing Implementation Agents

When spawning agents on worktrees to fix findings, use this template. The file lists come from the action items above — the orchestrator already identified them during synthesis.

```
First, read ~/.claude/AGENT.md for instructions.

You are fixing: {finding description from action item}

Files to READ (context only — do not modify):
{exact file list from action item}

Files to EDIT:
{exact file list from action item}

IMPORTANT: Do NOT explore the codebase. Do NOT read files not listed above.
Do NOT run grep/glob to discover files. Everything you need is listed here.

Fix: {specific fix instructions from the recommended fix}

After making changes, run the relevant test file(s):
{test file path if known, otherwise "npx vitest run tests/unit/<relevant>.test.ts"}
```

### Why this matters

Each implementation agent on a worktree starts with a blank context. Without explicit file lists, it will spend ~30K tokens exploring the codebase to figure out what to change — the same exploration the review agents already did. By passing the file lists from the synthesised action items, each implementation agent costs ~10K tokens instead of ~50K.

## Rules

- Do NOT review code yourself — delegate only
- Do NOT edit any files — read-only pipeline
- Do NOT include e2e-tests — it's a builder, not an auditor
- ALWAYS run Step 0.5 (pre-flight brief) before launching review agents
- In star mode: do NOT launch agents sequentially
- In ring mode: do NOT launch agents in parallel — each needs the previous report
- Model assignments: architect and security on `"sonnet"`, style/compliance/production on `"haiku"`
- Implementation agents MUST receive explicit file lists — never let them explore
